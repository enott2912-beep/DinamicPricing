# Алгоритм динамического ценообразования для ритейла (MVP)

Данный проект представляет собой профессиональную MVP-систему для автоматизации ценообразования в ритейле. Система объединяет генерацию синтетических данных, аналитические модели (эвристики и регрессию) и интерактивный интерфейс для управления сценариями продаж.

```text
DinamicPricing/
├── model/
```

## 🚀 Инструкция по запуску

1. **Окружение**: Рекомендуется Python 3.10+.
2. **Установка**:
   ```bash
   pip install -r requirements.txt
   ```
3. **Генерация данных**:

## ✨ Ключевые возможности

*   **Сценарная симуляция (Scenario Forecasting)**: Прогноз выручки на 7-30 дней вперед с учетом выбранного периода обучения. 
*   **Интеллектуальный конкурент**: В симуляции заложена логика реакции конкурента на ваши действия (реактивный рынок).
*   **Двойная модель**: 
    *   *Rule-based*: Быстрая реакция на демпинг и отсутствие спроса.
    *   *Regression-based*: Поиск математического оптимума для максимизации выручки.
*   **Аналитика**: Интерактивный календарь для выбора периодов обучения и визуализация эластичности.

## 🧠 Методология

### 1. Линейная эластичность спроса
Для моделирования зависимости спроса от цены в MVP используется линейная регрессия:
$$Sales = A - B \cdot Price + \epsilon$$
Где $B$ — коэффициент чувствительности спроса к цене.

### 2. Математическая оптимизация выручки
Целевая функция выручки ($Rev$):
$$Rev = P \cdot (A - B \cdot P) = AP - BP^2$$
Чтобы найти цену ($P$), максимизирующую выручку, найдем точку, где производная функции равна нулю:
$$\frac{dRev}{dP} = A - 2BP = 0 \implies P_{opt} = \frac{A}{2B}$$

### 3. Цикл "Наблюдение — Обучение — Прогноз"
Система реализует принцип автопилота: каждые 3 дня симуляции модель переобучается на вновь поступивших данных, адаптируясь к рыночным изменениям.

## 🚀 Future Roadmap (Enterprise Scale)

Для масштабирования текущего MVP до уровня полноценного коммерческого Enterprise-решения для крупного ритейла, в проект планируется заложить следующие архитектурные и бизнес-изменения:

### 1. Переход на ML-модели причинно-следственного вывода (Causal Inference)
Базовая линейная регрессия заменяется на лог-линейные модели или градиентный бустинг (LightGBM/XGBoost) с инструментами Causal Inference. Это необходимо для очистки влияния цены на спрос от внешних факторов (сезонность, каннибализация, наличие промо-акций, дни недели).

### 2. Оптимизация Маржи (Profit Optimization)
Вместо максимизации сырой выручки ($P \times Q$), целевой функцией становится **Прибыль**: $(P - COGS) \times Q$, где $COGS$ — себестоимость товара. 
Оптимизация цены будет происходить (например, через `SciPy.optimize`) с учетом строгих **бизнес-ограничений (Constraints)**: *минимальная маржинальность*, *РРЦ*, *максимально допустимый шаг изменения цены*.

### 3. Настраиваемый Rule Engine
Отказ от захардкоженных в коде "if-else" бизнес-правил. Внедрение базы данных и UI-редактора, где категорийный менеджер может сам конструировать правила (например, *"Если наш конкурент Агрессор опустил цену, держать разницу не более 2%"*).

### 4. Векторизация и Иерархические байесовские модели (Highload)
* **Иерархия:** Данные в ритейле собираются по уровням (`Формат магазина -> Локация -> Категория -> SKU`). Если по конкретному товару в данном магазине мало продаж (редкий спрос), модель "одалживает" эластичность у всей категории этого региона.
* **Скорость:** Перевод симуляции с циклов `for` на полные векторизованные умножения матриц (`NumPy/PyArrow`) или `Apache Spark` для переваривания сотен тысяч SKU за минуты.
* **OOS-фактор (Out of Stock):** Интеграция данных об остатках для исключения нулевых продаж из обучающей выборки модели.

### 5. Кросс-эластичность (Каннибализация)
Учет матрицы влияния всех товаров друг на друга. Снижение цены на акционный товар (например, пиво) должно генерировать дополнительный спрос на сопутствующие (чипсы/орешки), а не просто считаться независимо.
