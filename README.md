# Алгоритм динамического ценообразования для ритейла (MVP)

🚀 **Демо:** [https://dinamicpricing.streamlit.app](https://dinamicpricing.streamlit.app)

Данный проект представляет собой профессиональную MVP-систему для автоматизации ценообразования в ритейле. Система объединяет генерацию синтетических данных, аналитические модели (эвристики и регрессию) и интерактивный интерфейс для управления сценариями продаж.

## 🚀 Инструкция по запуску

1. **Окружение**: Рекомендуется Python 3.10+.
2. **Установка**:
   ```bash
   pip install -r requirements.txt
   ```
3. **Unit-тесты** (опционально):
   ```bash
   pip install -r requirements-dev.txt
   pytest
   ```
   Сводная таблица «тест → функция»: [TESTS.md](TESTS.md).
4. **Генерация данных**:
   ```bash
   python generator/generate_data.py
   ```
   Полностью перезаписывает `data/sales_history.csv` и сбрасывает `data/predict_sales.csv` (пустой заголовок), чтобы старые прогнозы не смешивались с новой историей.

   **Как устроена генерация цен.** Вместо независимого случайного шума каждый день (прежняя реализация) цена теперь держится на одном уровне в течение сегмента случайной длины (7–21 день), после чего скачком переходит на новый уровень. Это даёт истории периодов, когда цена стабильна, и периодов, когда она меняется — регрессия может выучить эластичность по истории одного товара. Амплитуда скачков разная по товарам (например, `Кофе: ±25%`, `Молоко и Хлеб: ±8%`) — товары с низкой ценовой эластичностью колеблются меньше, что ближе к реальному рынку.

5. **Запуск приложения**:
   ```bash
   streamlit run ui/streamlit_app.py
   ```

## 🗂 Хранение данных по сессиям (без ЛК)

Система поддерживает анонимные сессии без регистрации пользователя:

- для каждой сессии создается `session_id` (UUID);
- `session_id` сохраняется в URL (`?sid=...`);
- данные изолируются по сессии в `data/app_state.db` (SQLite).

Что хранится отдельно для каждой сессии:

- пользовательские правила Rule Engine;
- история продаж (`sales_history`);
- прогнозные строки симуляции (`predict`/forecast);
- режим генерации (`baseline`/`experimental`).

TTL сессии: по умолчанию 14 дней без активности. Просроченные сессии удаляются автоматически.

### Управление сессией в UI

В боковой панели доступны:

- `🆕 Новая сессия` — создаёт новый `session_id` и полностью изолированный контекст;
- `🧹 Очистить` — очищает правила и данные текущей сессии (без удаления кода приложения).

## 🧭 Режимы работы

В приложении доступны два режима:

- **Проверенный (`baseline`)** — классический контур с методами `rules + linear regression` и базовым генератором синтетических данных.
- **Тестовый (`experimental`)** — нелинейный контур с моделью `LightGBM` и отдельным нелинейным генератором данных.

> Важно: при переключении режима необходимо заново сгенерировать историю продаж, чтобы выбранный режим и тип данных совпадали.

## ✨ Ключевые возможности

*   **Сценарная симуляция (Scenario Forecasting)**: Прогноз выручки на 7-30 дней вперед с учетом выбранного периода обучения. 
*   **Интеллектуальный конкурент**: В симуляции заложена логика реакции конкурента на ваши действия (реактивный рынок).
*   **Настраиваемый Rule Engine (Динамические правила)**: Отказ от захардкоженных в коде бизнес-правил. Внедрен JSON-конфигуратор и визуальный UI-редактор во вкладке "Симуляция". Менеджер может сам "на лету" конструировать правила (название, условие, действие), удалять неактуальные и моментально откатывать изменения через систему истории, тестируя эффект сценариев на графике.
*   **Анонимные сессии на SQLite**: Данные (правила, история, прогнозы) изолированы между пользователями даже без личного кабинета.
*   **Два режима моделей**:
    *   *Rule-based*: Умная оценка истории на базе динамических правил и реакция на рынок.
    *   *Regression-based (Linear)*: Поиск математического оптимума для максимизации выручки по истории SKU.
    *   *Nonlinear (LightGBM, тестовый режим)*: Нелинейная оценка спроса с расширенными признаками.
*   **Интерактивная аналитика (Plotly)**: Все дашборды и графики (динамика цен, зависимость спроса, симуляция 365+ дней вперед) поддерживают зумирование, всплывающие подсказки и скрытие слоев благодаря переходу на Plotly.

## 🧠 Методология

### 1. Линейная эластичность спроса
Для моделирования зависимости спроса от цены в MVP используется линейная регрессия:
$$Sales = A - B \cdot Price + \epsilon$$
Где $B$ — коэффициент чувствительности спроса к цене. В реализации оценка $(A, B)$ — обычная МНК-регрессия (`scikit-learn`); член $\epsilon$ отражает остаток модели, в табличных прогнозах используется оценка без явного добавления случайного $\epsilon$.

### 2. Математическая оптимизация выручки
Целевая функция выручки ($Rev$):
$$Rev = P \cdot (A - B \cdot P) = AP - BP^2$$
Чтобы найти цену ($P$), максимизирующую выручку, найдем точку, где производная функции равна нулю:
$$\frac{dRev}{dP} = A - 2BP = 0 \implies P_{opt} = \frac{A}{2B}$$

### 3. Что именно прогнозируется в интерфейсе

*   **Эвристика (вкладка «Рекомендации»):** после выбора цены правилами выручка оценивается по **заданным в коде параметрам `PRODUCTS`** (базовая цена, эластичность, базовый спрос), а не по регрессии. Колонка «Δ к факту, %» сравнивает этот прогноз с **фактической выручкой последнего дня по SKU**; при правиле `hold` цена не меняется, но Δ может быть ненулевым, если факт расходится с формулой `PRODUCTS`.
*   **Регрессия (рекомендации):** прогноз выручки — из той же линии **Sales ≈ A − B·Price**, что и для оптимума; сравнение с фактом — так же по последнему дню.

### 4. Симуляция и согласованность с моделями

*   **Период обучения** совпадает с выбранным в календаре на вкладке «Обзор» диапазоном: в `simulate` передаётся отфильтрованный `DataFrame`.
*   **Параметры симуляции (вкладка «Симуляция»):** для методов `regression` и `lightgbm` задаются **интервал переобучения** («Переобучать каждые N дней», 1–30), **скользящее окно обучения** (дней истории) и **лимит дневного изменения цены (%)**. На шаге 0 модель всегда переобучается; далее линейная регрессия и LightGBM заново подгоняются на накопленной истории (с учётом окна) в соответствии с выбранным интервалом. Между переобучениями для `regression` целевая цена берётся с последнего переобучения (пока не сработает следующий шаг переобучения).
*   **Метод `regression`:** на каждом шаге переобучения по скользящему окну заново оцениваются коэффициенты **Sales ≈ A − B·Price**; в обучение попадают только дни **без OOS** (`is_oos`) и с **положительными** продажами. Надёжность фита проверяется не только по знаку коэффициента, но и по **R² ≥ 0.25** (см. `MIN_R2_FOR_RELIABLE_FIT` в `model/pricing.py`) — раньше проверка только по знаку пропускала модели с отрицательным R² на отложенных данных (см. раздел «Известные ограничения» ниже). Итоговые цена и спрос — не бинарное переключение «регрессия/эвристика», а **смешивание (blending)** по непрерывному весу доверия `w = clip(R² / 0.25, 0, 1)`: `итог = w·регрессия + (1−w)·эвристика(PRODUCTS + конкурент)`. Это убирает скачок прогноза на границе порога надёжности.
*   **Метод `rules`:** на каждом дне цена пересчитывается эвристикой по последнему состоянию; спрос моделируется формулой `PRODUCTS` и динамикой конкурента.
*   **Метод `lightgbm` (тестовый режим):** отдельный нелинейный генератор данных, бустинг `LightGBM` с диагностикой по объёму и вариативности цен; переобучение по тому же расписанию и окну, что и у регрессии. Диагностика включает несколько этапов:

    - **Проверка объёма и вариативности цены:** история сущности должна содержать не менее `LGBM_MIN_ROWS = 60` строк, не менее `LGBM_MIN_UNIQUE_PRICES = 8` уникальных цен, а коэффициент вариации цены (`CV = std/mean`) — быть не ниже `LGBM_MIN_PRICE_CV = 0.03`. Порог задан в относительных единицах (а не абсолютном std), чтобы не отбрасывать данные с малой абсолютной ценой, но существенной относительной вариативностью (см. валидацию на датасете Avocado Prices, Kaggle). Если любая из проверок не пройдена — модель не обучается, `trust_weight = 0`.

    - **Holdout-валидация после обучения:** последние `LGBM_HOLDOUT_RATIO = 20%` хронологии откладываются для финальной оценки. На этом отрезке считаются `R²_holdout` и `MAE_ratio = MAE / mean_sales`. Модель считается надёжной (`reliable = True`), если `R²_holdout ≥ LGBM_R2_THRESHOLD = 0.20` и `MAE_ratio ≤ LGBM_MAE_RATIO_MAX = 0.50`. Вес доверия `trust_weight = min(1.0, R²_holdout / 0.60)` плавно снижает вклад модели при ухудшении holdout-метрик.

    - **Экстраполяционный вес:** при расчёте прогноза спроса для цены за пределами исторического диапазона `[p_lo, p_hi]` (5-й и 95-й перцентили) вес модели линейно падает до нуля на расстоянии `LGBM_EXTRAP_DECAY_WIDTH = 10%` от полуширины диапазона. Если `R²_holdout` ниже порога — вес модели обнуляется полностью.

    - **Автотюнинг гиперпараметров:** при достаточном объёме истории (`≥ LGBM_TUNE_MIN_ROWS = 90` строк после очистки) запускается `RandomizedSearchCV` с `n_iter = LGBM_TUNE_N_ITER = 4` и `TimeSeriesSplit(3)` по распределениям `n_estimators`, `num_leaves`, `learning_rate`, `min_child_samples`. При нехватке данных или любой ошибке тюнинга тихо откатывается на фиксированный набор по умолчанию (`LGBM_DEFAULT_PARAMS`: `n_estimators=180`, `learning_rate=0.05`, `num_leaves=31`, `min_child_samples=20`). Малое число итераций (`n_iter=4`) — осознанный компромисс скорости, так как тюнинг вызывается на каждое переобучение в симуляции для каждой сущности.

## ⚠️ Известные ограничения и валидация на внешних данных

Пайплайн обучения проверялся не только на собственном синтетическом генераторе, но и на двух внешних открытых датасетах — цифры ниже не предположены, а получены прямым прогоном прод-функций (см. `tests/test_external_dataset_sanity.py`, `tests/test_lgbm_thresholds_scale_invariant.py`, а также `notebooks/03_external_data_validation.ipynb`).

### Retail Price Optimization (Kaggle) — разреженный сценарий

Датасет `tests/fixtures/external_retail_price_sample.csv`: 52 SKU, месячные данные (5–20 точек на товар). Сценарий, где линейная регрессия ожидаемо слаба — точек мало.

*   **Критерий надёжности регрессии был слишком слабым.** Проверка только по знаку коэффициента помечала **~65.4%** товаров как «надёжные», хотя их out-of-sample R² был сильно отрицательным. После добавления порога `R² ≥ MIN_R2_FOR_RELIABLE_FIT = 0.25` доля упала до **~19.2%** — это ближе к реальному качеству.
*   **Абсолютное качество на разреженных данных низкое, и это ожидаемо.** На 5–20 месячных точках добиться высокого R² объективно сложно — это ограничение объёма данных, а не повод считать модель «сломанной».
*   Blending (непрерывный вес `w = clip(R² / 0.25, 0, 1)`) ведёт себя корректно: для слабых фитов вес дробный, итоговая цена — плавная смесь регрессии и эвристики.

### Avocado Prices (Kaggle) — частый сценарий

Датасет `tests/fixtures/external_avocado_weekly_sample.csv`: 3 региона × 2 типа = 6 сущностей, 169 еженедельных точек на сущность. Цена ~$1–1.5, std ~$0.20 (CV ≈ 15%). Сценарий, где LightGBM может реально показать себя.

*   **Порог вариативности цены был масштабно-зависимым.** Старый порог `LGBM_MIN_PRICE_STD = 1.0` (абсолютный std) отбрасывал **все 100%** сущностей ещё до обучения — цена в долларах физически не могла иметь std ≥ 1.0. После перехода на относительный порог `LGBM_MIN_PRICE_CV = 0.03` (коэффициент вариации) модель начала обучаться, и holdout-валидация реально запустилась.
*   **Holdout-валидация LightGBM дала измеримый результат.** Доля сущностей, признанных надёжными (`R²_holdout ≥ LGBM_R2_THRESHOLD = 0.20` и `MAE_ratio ≤ LGBM_MAE_RATIO_MAX = 0.50`), составила **~46.3%**. Это контрастирует с 0% при старом пороге — не потому что модель стала лучше, а потому что она вообще получила шанс обучиться.
*   **Важное допущение:** в датасете avocado нет родной цены конкурента — она заменена на среднюю цену по другим регионам того же типа, а `cogs` — на грубую прокси `price × 0.6`. Цифры reliable стоит воспринимать как иллюстрацию работы механизмов, а не как точную меру качества для реального avocado-рынка.

### Разрыв между ноутбуками и продом

В ноутбуках `notebooks/01_model_training_and_evaluation.ipynb` и `02_multi_sku_evaluation.ipynb` `GridSearchCV`/`RandomizedSearchCV` подбирают гиперпараметры LightGBM с полным перебором или `n_iter=25`. В проде `_tune_lgbm_params()` использует `n_iter=4` — это осознанный компромисс скорости: тюнинг вызывается на каждое переобучение для каждой сущности в симуляции. Совпадение лучших параметров не гарантировано, и это нормально.

## 🚀 Future Roadmap (Enterprise Scale)

Ниже — направления развития поверх текущего MVP. Часть «классического» roadmap уже заложена в коде в упрощённом виде.

**Уже есть в MVP (кратко):** тестовый контур **LightGBM** и нелинейная генерация истории; в симуляции и рекомендациях учитываются **прибыль и COGS**; в данных и обучении используется признак **out-of-stock** (`is_oos`); **лимит дневного шага цены** и **периодическое переобучение** в симуляции.

### 1. Причинно-следственный вывод (Causal Inference)
Расширить оценку эффекта цены на спрос методами causal inference (например, лог-линейные спецификации, двойной машинный обучающийся (DML), веса по пропенсити), чтобы отделить влияние цены от сезонности, промо, дня недели и прочих смещений. Текущие регрессия и LightGBM остаются корреляционными моделями прогноза.

### 2. Оптимизация под жёсткими ограничениями
Сейчас оптимум цены ищется аналитически (линейная ветка) или перебором кандидатов (LightGBM) с полом по COGS и лимитом шага в симуляции. Следующий шаг — единая **многокритериальная** постановка с пакетом ограничений (минимальная маржа, РРЦ, минимальный/максимальный сток, связки SKU), в т.ч. через численные солверы (`SciPy.optimize` и аналоги).

### 3. Highload: иерархия, векторизация, распределённые расчёты
* **Иерархические / байесовские модели:** «заимствование» эластичности и профилей спроса с уровня категории/региона на редкие SKU.
* **Скорость:** векторизация симуляции (`NumPy`/`PyArrow`) или вынесение на **`Apache Spark`** для очень больших сеток магазин × SKU.
* **OOS:** углубление учёта остатков и сценариев потерь продаж (не только флаг в обучающей выборке).

### 4. Кросс-эластичность (каннибализация)
Матрица взаимного влияния товаров: совместное изменение спроса при ценовых акциях и кросс-продажах, а не только независимые SKU.
